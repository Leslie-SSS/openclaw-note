# 网站备份与灾难恢复

## 一、理论框架

### 1.1 备份与灾难恢复概述

网站备份与灾难恢复是保障独立站业务连续性的核心组成部分。在数字化时代，数据丢失可能导致严重的经济损失和品牌信誉受损。建立完善的备份与灾难恢复体系，是每个独立站运营者的必修课。

**备份的核心价值**：
- **业务连续性保障**：确保在故障发生时能够快速恢复
- **数据资产保护**：防止数据丢失造成不可挽回的损失
- **合规性要求**：满足行业监管和法律法规的要求
- **风险降低**：减少因数据丢失导致的潜在损失

### 1.2 备份类型与策略

**按备份内容分类**：

| 备份类型 | 内容 | 优点 | 缺点 | 适用场景 |
|----------|------|------|------|----------|
| 完全备份 | 所有数据 | 恢复简单 | 时间长、占用大 | 周期性基础备份 |
| 增量备份 | 变化数据 | 速度快、占用小 | 恢复复杂 | 日常频繁备份 |
| 差异备份 | 相对基准的变化 | 速度适中 | 占用逐渐增大 | 中等频率备份 |
| 镜像备份 | 实时复制 | RPO接近零 | 成本高 | 关键业务系统 |

**按备份方式分类**：

- **冷备份**：离线备份，恢复时间较长但安全性高
- **热备份**：在线备份，系统运行时进行，对业务影响小
- **温备份**：半在线备份，部分服务可用

### 1.3 RTO与RPO概念

**关键指标定义**：

- **RPO（Recovery Point Objective）**：恢复点目标，指可接受的最大数据丢失量，以时间衡量
- **RTO（Recovery Time Objective）**：恢复时间目标，指系统从故障到恢复正常运行的最大可接受时间

**不同业务场景的RTO/RPO要求**：

| 业务类型 | 建议RPO | 建议RTO | 备份策略 |
|----------|---------|---------|----------|
| 高交易电商 | <1小时 | <1小时 | 实时复制+增量备份 |
| 普通电商 | <4小时 | <4小时 | 每日备份+增量同步 |
| 内容展示站 | <24小时 | <24小时 | 每日完全备份 |
| 低频访问站 | <72小时 | <72小时 | 每周完全备份 |

### 1.4 灾难恢复等级

根据GB/T 20988-2007《信息系统灾难恢复规范》，灾难恢复分为6个等级：

| 等级 | 描述 | RTO | RPO | 特征 |
|------|------|-----|-----|------|
| 1级 | 基本支持 | >72小时 | 24-72小时 | 数据介质异地存放 |
| 2级 | 备用场地支持 | 24-72小时 | 24-72小时 | 备用场地+数据备份 |
| 3级 | 电子传输 | 12-24小时 | 数小时 | 网络传输+部分设备 |
| 4级 | 电子传输+设备就绪 | 数小时 | 数小时 | 热站设备就绪 |
| 5级 | 实时传输 | 分钟级 | 分钟级 | 数据实时复制 |
| 6级 | 数据零丢失 | 分钟级 | 0 | 自动故障切换 |

### 1.5 3-2-1备份原则

业界广泛认可的备份黄金法则：

- **3份副本**：至少保留3份数据副本（1份原件+2份备份）
- **2种介质**：使用至少2种不同的存储介质
- **1份异地**：至少1份备份存储在异地

**扩展3-2-1-1-0原则**：
- **1份离线**：至少1份备份保持离线/隔离状态
- **0错误**：定期验证备份，确保备份可用且无错误

### 2.2 备份实施方案

**步骤一：数据库备份配置**

```bash
#!/bin/bash
# MySQL数据库备份脚本

# 配置变量
DB_HOST="localhost"
DB_USER="backup_user"
DB_PASS="your_password"
DB_NAME="your_database"
BACKUP_DIR="/backup/database"
DATE=$(date +%Y%m%d_%H%M%S)
RETENTION_DAYS=30

# 创建备份目录
mkdir -p $BACKUP_DIR

# 执行完全备份
mysqldump -h$DB_HOST -u$DB_USER -p$DB_PASS --single-transaction \
  --routines --triggers --events $DB_NAME | gzip > \
  $BACKUP_DIR/full_backup_$DATE.sql.gz

# 记录日志
echo "$(date) - Full backup completed: full_backup_$DATE.sql.gz" \
  >> $BACKUP_DIR/backup.log

# 清理过期备份
find $BACKUP_DIR -name "*.sql.gz" -type f -mtime +$RETENTION_DAYS -delete

echo "$(date) - Cleaned backups older than $RETENTION_DAYS days" \
  >> $BACKUP_DIR/backup.log
```

**步骤二：文件备份配置**

```bash
#!/bin/bash
# 文件同步备份脚本（使用rsync）

# 配置变量
SOURCE_DIR="/var/www/html"
REMOTE_HOST="backup-server.example.com"
REMOTE_USER="backup"
REMOTE_DIR="/backup/files"
LOG_FILE="/var/log/backup_sync.log"

# 执行同步
rsync -avz --delete \
  --exclude='*.log' \
  --exclude='cache/*' \
  --exclude='tmp/*' \
  -e "ssh -i /home/backup/.ssh/id_rsa" \
  $SOURCE_DIR $REMOTE_USER@$REMOTE_HOST:$REMOTE_DIR 2>&1 | \
  tee -a $LOG_FILE

# 记录结果
if [ ${PIPESTATUS[0]} -eq 0 ]; then
    echo "$(date) - File sync completed successfully" >> $LOG_FILE
else
    echo "$(date) - File sync FAILED" >> $LOG_FILE
    # 发送告警
    echo "Backup failed at $(date)" | mail -s "Backup Alert" admin@example.com
fi
```

#!/bin/bash

### 2.3 备份验证与测试

**步骤一：自动化验证脚本**

```bash
#!/bin/bash
# 备份完整性验证脚本

BACKUP_FILE=$1
TEMP_DIR="/tmp/backup_verify"

# 检查文件是否存在
if [ ! -f "$BACKUP_FILE" ]; then
    echo "Error: Backup file not found"
    exit 1
fi

# 检查文件大小
FILE_SIZE=$(stat -f%z "$BACKUP_FILE" 2>/dev/null || stat -c%s "$BACKUP_FILE")
if [ $FILE_SIZE -lt 1000 ]; then
    echo "Error: Backup file too small"
    exit 1
fi

# 验证压缩文件完整性
if gzip -t "$BACKUP_FILE" 2>/dev/null; then
    echo "Gzip integrity check: PASSED"
else
    echo "Gzip integrity check: FAILED"
    exit 1
fi

# 验证SQL备份（如果是数据库备份）
mkdir -p $TEMP_DIR
gunzip -c "$BACKUP_FILE" > $TEMP_DIR/test.sql
if grep -q "CREATE TABLE" $TEMP_DIR/test.sql && \
   grep -q "INSERT INTO" $TEMP_DIR/test.sql; then
    echo "SQL content check: PASSED"
else
    echo "SQL content check: FAILED"
fi

# 清理
rm -rf $TEMP_DIR

echo "Backup verification completed"
```

### 4.2 负面案例：某独立站数据丢失事件

**背景**：
- 年销售额约50万美元的小型独立站
- 使用共享主机
- 无专职IT人员

**事件经过**：

1. **初始问题**：
   - 主机商服务器硬盘故障
   - 发现主机商的"每日备份"实际上只保留7天
   - 最近3天的数据完全丢失

2. **损失评估**：
   - 丢失约150笔订单数据
   - 丢失约500条新用户注册
   - 丢失约2000个产品更新

3. **后续处理**：
   - 手动恢复部分数据（从邮件记录）
   - 向客户道歉并提供补偿
   - 预计损失$15,000（含补偿和品牌损失）

**教训**：
- 不能完全依赖主机商备份
- 需要建立独立的备份机制
- 应测试备份恢复流程
