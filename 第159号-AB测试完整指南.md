## 一、A/B测试概述

### 1.1 A/B测试定义

A/B测试是一种对比实验方法，将用户随机分配到两个或多个版本，通过数据分析确定哪个版本能带来更好的结果。

### 1.2 A/B测试价值

| 价值维度 | 具体收益 | 典型提升 |
|---------|---------|---------|
| 转化优化 | 提升转化率 | 10-50% |
| 收入增长 | 提升GMV | 5-30% |
| 用户体验 | 降低跳出率 | 15-40% |
| 风险降低 | 避免错误决策 | - |
| 数据决策 | 消除主观判断 | - |
| 持续改进 | 积累优化经验 | - |

### 1.3 测试类型矩阵

```
A/B测试类型:

┌────────────────────────────────────────────────┐
│                                               │
│   A/B测试         A/B/n测试        多变量测试  │
│   2个版本         3+个版本         多因素组合  │
│                                               │
├────────────────────────────────────────────────┤
│                                               │

```

## 三、样本量计算

### 3.1 样本量计算公式

```
样本量计算核心公式:

n = (Z_α/2 + Z_β)² × 2 × p × (1-p)
    ─────────────────────────────────
              (p × MDE)²

其中:
n = 每组所需样本量
Z_α/2 = 显著性水平对应Z值 (95%置信 = 1.96)
Z_β = 统计功效对应Z值 (80%功效 = 0.84)
p = 基准转化率
MDE = 最小可检测效应 (Minimum Detectable Effect)

简化公式:
n = 16 × p × (1-p) / (p × MDE)²

示例计算:
基准转化率 p = 5%
预期提升 MDE = 20% (从5%提升到6%)
显著性 α = 5%
统计功效 1-β = 80%

n = 16 × 0.05 × 0.95 / (0.05 × 0.20)²
n = 0.76 / 0.0001
n = 7,600 (每组)

总样本量 = 7,600 × 2 = 15,200
```

### 3.3 测试周期计算

```
测试周期计算:

测试天数 = 所需样本量 ÷ 日均流量 × 分组数

示例:
- 每组所需样本: 7,600
- 日均流量: 2,000
- 测试版本: 2个

测试天数 = 7,600 × 2 ÷ 2,000 = 7.6天

考虑因素:
├── 业务周期: 至少覆盖1-2个完整周期
├── 行为差异: 工作日vs周末
├── 外部因素: 避免节假日、大促期间
└── 安全边际: 增加10-20%时间

最小测试周期建议:
├── 最低: 7天
├── 推荐: 14天
└── 复杂测试: 21-28天
```

## 四、测试设计最佳实践

### 4.1 随机分配设计

```python
# 用户分组示例代码

import hashlib

def assign_group(user_id, experiment_id, num_groups=2):
    """
    基于用户ID的一致性随机分组
    
    参数:
    user_id: 用户唯一标识
    experiment_id: 实验唯一标识
    num_groups: 分组数量
    
    返回:
    组别 (0 到 num_groups-1)
    """
    # 创建唯一标识符
    identifier = f"{user_id}_{experiment_id}"
    
    # 使用MD5生成哈希值
    hash_value = hashlib.md5(identifier.encode()).hexdigest()
    
    # 转换为整数并取模
    bucket = int(hash_value, 16) % num_groups
    
    return bucket

# 示例使用
user_id = "user_12345"
experiment_id = "checkout_button_test_202401"
group = assign_group(user_id, experiment_id)
print(f"User {user_id} assigned to group: {group}")
```

## 五、统计分析方法

### 5.1 统计显著性

```
统计显著性计算:

假设检验:
├── H0 (零假设): A版本 = B版本
├── H1 (备择假设): A版本 ≠ B版本
└── 显著性水平 α: 通常设为0.05

P值解读:
├── p < 0.05: 结果显著，拒绝H0
├── p < 0.01: 结果非常显著
└── p >= 0.05: 结果不显著，无法拒绝H0

Z检验公式 (两比例):

Z = (p₁ - p₂) / √(p̂ × (1-p̂) × (1/n₁ + 1/n₂))

其中:
p₁, p₂ = 两个版本的转化率
p̂ = 合并转化率 = (x₁ + x₂) / (n₁ + n₂)
n₁, n₂ = 两个版本的样本量

示例:
版本A: 10000样本, 500转化 (5%)
版本B: 10000样本, 600转化 (6%)

p̂ = (500 + 600) / (10000 + 10000) = 0.055
Z = (0.05 - 0.06) / √(0.055 × 0.945 × 0.0002)
Z = -0.01 / 0.0032 = -3.125

查Z表: p < 0.01
结论: 版本B显著优于版本A
```

### 5.2 置信区间

```
置信区间计算:

转化率差异的95%置信区间:

CI = (p₁ - p₂) ± Z_α/2 × √(p₁(1-p₁)/n₁ + p₂(1-p₂)/n₂)

示例:
版本A: 5.0% ± 0.43%
版本B: 6.0% ± 0.47%

差异置信区间: 
-1.47% 到 -0.53%

解读:
- 我们95%确信版本B比版本A高0.53%-1.47%
- 因为置信区间不包含0，差异显著

实用意义:
├── 效果大小评估
├── 业务价值估算
└── 决策风险量化
```

### 5.3 统计功效

```yaml
统计功效 (Statistical Power):

定义:
  当真实存在差异时，检测到该差异的概率
  
推荐值:
  - 最低: 80%
  - 推荐: 85-90%
  
影响因素:
  样本量:
    - 样本越大，功效越高
  
  效应大小:
    - 差异越大，功效越高
  
  显著性水平:
    - α越大，功效越高 (但也增加假阳性)
  
  方差:
    - 方差越小，功效越高

功效不足的后果:
  - 真实差异存在但未能检测
  - 浪费时间和资源
  - 错过优化机会
```

### 5.4 多重比较校正

```
多重比较问题:

问题:
├── 同时测试多个版本时，假阳性率增加
├── 3个版本对比，整体α > 0.05
└── 需要校正显著性水平

校正方法:

1. Bonferroni校正
   α_adj = α / n (n为比较次数)
   
   示例: 3个版本，2次比较
   α_adj = 0.05 / 2 = 0.025

2. Holm-Bonferroni校正
   - 按p值排序
   - 渐进式校正
   - 比Bonferroni更强大

3. Tukey HSD
   - 适用于多组比较
   - 考虑所有配对比较
   - 提供置信区间

建议:
├── 2-3个版本: 使用Bonferroni
├── 4+个版本: 使用Holm或Tukey
└── 严格场景: 降低α到0.01
```

### 7.3 结果解读注意事项

```
常见陷阱:

1. 提前停止测试
   问题: 样本量不足，结果不可靠
   解决: 坚持达到预定样本量

2. 多次查看结果
   问题: 增加假阳性率
   解决: 预设定检查时间点

3. 忽略置信区间
   问题: 只看点估计，忽略不确定性
   解决: 始终报告置信区间

4. 过度分层
   问题: 子样本太小，统计功效不足
   解决: 分层分析仅作参考

5. 忽略实际意义
   问题: 统计显著但业务不显著
   解决: 评估实际业务价值

6. 辛普森悖论
   问题: 分组趋势与整体趋势相反
   解决: 检查组间均衡性
```

## 九、高阶测试策略

### 9.1 多变量测试

```
多变量测试 (MVT):

定义: 同时测试多个变量的组合效果

示例:
变量1: 按钮颜色 (红/蓝)
变量2: 按钮文字 (购买/立即购买)
变量3: 价格显示 (含税/不含税)

组合: 2 × 2 × 2 = 8个版本

优势:
├── 测试变量交互效应
├── 找出最优组合
└── 效率高于顺序测试

劣势:
├── 样本量需求大
├── 分析复杂度高
└── 实施难度大

适用场景:
├── 高流量页面
├── 多元素页面
└── 复杂决策因素
```

### 9.2 序列测试

```yaml
序列测试 (Sequential Testing):

定义: 允许在达到预设条件时提前停止测试

方法:
  O'Brien-Fleming:
    - 早期停止门槛高
    - 后期门槛降低
    - 保持整体α水平
  
  Pocock:
    - 各阶段门槛相同
    - 更容易提前停止
    - 需要更大样本量

优势:
  - 可提前发现显著结果
  - 节省时间和资源
  
劣势:
  - 假阳性风险
  - 需要严格方法论

实施建议:
  - 预设定检查时间点
  - 使用校正方法
  - 记录所有检查
```

### 9.3 贝叶斯A/B测试

```
贝叶斯方法:

核心思想:
├── 转化率是随机变量
├── 使用先验分布
├── 根据数据更新后验
└── 计算概率而非p值

计算公式:
P(B > A) = ∫∫ I(p_B > p_A) × P(p_A|data) × P(p_B|data) dp_A dp_B

优势:
├── 结果直观 (概率解释)
├── 可随时检查结果
├── 处理小样本更好
└── 自然融入先验知识

劣势:
├── 计算复杂
├── 先验选择主观
└── 解释需要统计知识

适用场景:
├── 低流量场景
├── 需快速决策
├── 有先验知识
└── 复杂模型
```

## 十、行业案例分析

### 10.1 电商案例

```
案例: 产品页CTA优化

背景:
- 产品页转化率: 3.5%
- 日均流量: 20000
- 目标: 提升转化率到4%+

假设:
如果将CTA按钮从"加入购物车"改为"立即抢购"
那么转化率提升15%
因为"立即抢购"制造紧迫感

测试设计:
- 对照组: "加入购物车" (蓝色按钮)
- 实验组: "立即抢购" (红色按钮)
- 样本量: 每组15000 (共30000)
- 测试周期: 14天

结果:
- 对照组: 3.52% (528/15000)
- 实验组: 4.15% (623/15000)
- 提升: 17.9%
- p值: 0.003
- 95% CI: [0.39%, 0.87%]

分层分析:
- 新用户提升: 25.3%
- 老用户提升: 8.7%
- 移动端提升: 22.1%
- 桌面端提升: 12.4%

结论:
- 结果统计显著
- 建议全量采用
- 移动端优先发布
```

### 10.2 SaaS案例

```
案例: 注册流程优化

背景:
- 注册转化率: 12%
- 主要流失点: 邮箱验证步骤
- 目标: 提升到15%

假设:
如果跳过邮箱验证，改为首次登录时验证
那么注册转化率提升25%
因为减少注册摩擦

测试设计:
- 对照组: 注册时验证邮箱
- 实验组: 首次登录时验证邮箱
- 样本量: 每组5000
- 测试周期: 21天

结果:
- 对照组注册率: 12.3% (615/5000)
- 实验组注册率: 16.8% (840/5000)
- 提升: 36.6%
- p值: <0.001

护栏指标:
- 有效邮箱比例:
  - 对照组: 98.5%
  - 实验组: 92.3%
- 首次登录完成率:
  - 对照组: 85.2%
  - 实验组: 78.6%

综合分析:
- 注册转化提升显著
- 但邮箱验证完成率下降
- 有效用户净增加: 22.1%

结论:
- 采纳实验组方案
- 优化后续验证提醒
- 监控长期留存
```
